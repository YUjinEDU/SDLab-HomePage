# 데이터 모델 및 DB 스키마 초안

상위 문서: [[홈페이지 계획]]

이 문서의 역할:

- 데이터 구조와 DB 테이블 초안을 정의하는 문서

## MVP 구현 상태

> **현재 MVP는 정적 데이터 파일(`frontend/src/data/*.ts`)을 사용하며, Supabase DB 연결은 Phase 2에서 진행 예정이다.**

### 정적 데이터 파일 목록

- `data/members.ts` — 6명의 멤버 데이터
- `data/research-areas.ts` — 3개 연구 분야
- `data/publications.ts` — 6편의 논문
- `data/patents.ts` — 3건의 특허 (`Publication[]` 타입, `type="patent"`)
- `data/projects.ts` — 4개 프로젝트
- `data/news.ts` — 4개 소식
- `data/contact.ts` — 연락처 정보
- `data/stats.ts` — `labStats`(연구실 통계), `groupLabels`(그룹 라벨)

### MVP에서 사용 중인 TypeScript 타입 정의

타입 정의는 `frontend/src/types/`에 위치한다.


### DB 스키마와의 차이점

- **특허**: 별도 테이블 없이 `Publication` 타입에 `type="patent"`으로 통합 처리
- **stats**: 원래 스키마에 없던 `data/stats.ts`가 추가됨 (홈 화면 통계 표시용)
- **조인 테이블**: MVP에서는 배열 필드(`memberIds`, `researchAreaIds` 등)로 관계를 표현. DB 전환 시 조인 테이블로 분리 예정
- **visibility/status 컬럼**: MVP에서는 사용하지 않음 (모든 데이터가 공개 상태)

- `published_at`

  - `draft`, `published`, `archived`

운영 원칙:

- 홈 화면 Featured 영역은 `is_featured` 또는 별도 연결 테이블로 관리

## 핵심 테이블

### users (Phase 2 구현 예정)

> 인증 및 계정 관리는 Phase 2에서 Supabase Auth와 함께 구현 예정이다.

- 로그인 계정
- 역할과 계정 상태를 함께 관리

권장 필드:

- `id`
- `email`
- `name`
- `image`
- `role`
  - `member`, `professor`, `admin`
- `account_status`
  - `invited`, `active`, `inactive`, `suspended`, `archived`
- `member_id`
  - `members.id`와 선택적 연결
- `must_change_password`
- `is_active`
- `last_login_at`
- `invited_at`
- `activated_at`
- `created_at`
- `updated_at`

운영 원칙:

- `role`은 권한
- `account_status`는 계정 사용 상태
- 두 값은 분리해서 관리
- 관리자 발급 계정은 기본적으로 `account_status = invited`
- 첫 로그인 후 비밀번호 변경이 끝나면 `active`로 전환 가능

상태별 처리:

- `invited`
  - 첫 로그인 가능
  - 일반 내부 페이지 진입 전 비밀번호 변경 유도
- `active`
  - 정상 사용 가능
- `inactive`
  - 운영상 일시 비활성
- `suspended`
  - 로그인 차단 또는 강한 접근 제한
- `archived`
  - 졸업 또는 사용 종료 후 보관

- 공지, 수상, 행사, 논문 채택, 모집 글 통합 모델
- 카테고리, 요약, 본문, 행사 정보, 고정 여부 포함

- 내부 일정

- 내부 자료실

## 관계 원칙

- 멤버와 논문: 다대다
- 멤버와 프로젝트: 다대다
- 프로젝트와 논문: 다대다
- 연구 분야는 프로젝트 / 논문 / 데모와 연결
- 게시글은 프로젝트 / 논문과 선택적으로 연결
- `users.member_id -> members.id`는 선택적 1:1 연결

## 인증 / 계정 관리 관련 규칙

관련 문서:

- [[07 인증 및 계정관리 설계]]

원칙:

- 공개 회원가입은 두지 않는다
- 계정은 관리자 발급 방식으로 운영한다
- 모든 `member`가 `user`를 가지는 것은 아니다
- `admin`은 `member` 연결 없이 존재할 수 있다

초기 계정 생성 흐름:

1. 관리자 계정 생성
2. `role` 지정
3. 필요 시 `member_id` 연결
4. `account_status = invited`
5. 첫 로그인 후 `must_change_password = true`
6. 비밀번호 변경 완료 후 `account_status = active`

## Supabase 마이그레이션 계획 (Phase 2)

Phase 2에서 정적 데이터를 Supabase PostgreSQL로 이전한다.

마이그레이션 원칙:

- TypeScript 타입 정의를 기반으로 테이블 스키마를 확정한다
- 배열 필드(`memberIds`, `researchAreaIds` 등)는 조인 테이블로 분리한다
- `visibility`, `status` 등 운영 컬럼을 추가한다
- `data/*.ts`의 기존 데이터를 시드 데이터로 변환한다
- `lib/queries/`의 조회 함수를 Supabase 클라이언트 호출로 교체한다
