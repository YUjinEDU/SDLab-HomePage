## 구현 원칙

- Supabase client 설정 (`lib/db/client.ts`, `lib/db/server.ts`)
- Server Components에서 Supabase 쿼리 사용
- RLS(Row Level Security) 정책 적용
- 기존 정적 데이터와 동일한 타입 유지 (`types/*.ts` 재사용)
- `lib/queries/*.ts`로 데이터 접근 통일

## UI 원칙

- 과장된 스타트업 랜딩 페이지처럼 만들지 않는다
- 밝은 배경, 차분한 그린 포인트, 카드형 정보 구조 유지
- 검색/필터/UI 패턴은 페이지마다 다르게 만들지 않는다
- 빈 상태, 오류 상태, 검색 결과 없음 상태를 포함한다

## 콘텐츠 원칙

- 실제 연구실 정보는 [[10 연구실 기본 정보 정리]]를 우선 사용
- 정보가 없는 섹션은 억지로 채우지 않는다
- 문구는 과장하지 않고 연구실 홈페이지 톤을 유지한다

## 구현 우선순위

1. Supabase 설정 + 스키마 마이그레이션
2. lib/queries/_.ts 작성 (기존 data/_.ts 대체)
3. 기존 페이지 데이터 소스 교체 (data/ → queries/)
4. Supabase Auth 설정 + 로그인 페이지
5. middleware.ts 라우트 보호
6. /board 게시판 구현
7. (auth)/account 페이지

## 완성 판단 기준

- 모든 공개 페이지가 Supabase에서 데이터 조회
- 로그인/로그아웃 동작
- 역할별 라우트 보호 동작
- 게시판 CRUD 동작
- 기존 UI/UX 유지 (정적→동적 전환만)
