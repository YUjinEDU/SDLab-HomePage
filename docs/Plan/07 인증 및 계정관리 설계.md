# 인증 및 계정관리 설계

상위 문서: [[홈페이지 계획]]

## 구현 상태

**Phase 2 예정** — 공개 포털 MVP 완료 후 진행. 현재 인증 관련 코드는 미구현 상태이며, 아래 설계를 기반으로 Phase 2에서 Supabase Auth로 구현할 예정이다.

관련 문서:

- [[03 내부 포털 및 권한 설계]]
- [[04 데이터 모델 및 DB 스키마 초안]]
- [[06 Next.js 폴더 구조 및 App Router 설계]]

이 문서의 역할:

- 관리자 발급 계정 방식의 인증, 계정 상태, 계정 UX를 정의하는 문서

목적:

- 로그인 방식과 계정관리 범위를 분명히 한다.
- `user`와 `member`의 관계를 고정한다.
- 코딩 에이전트가 인증 흐름과 계정 페이지를 흔들리지 않고 구현하도록 한다.

## 기본 방향

- 인증은 `Supabase Auth`를 사용한다.
- 사용자 계정과 멤버 프로필은 분리된 엔티티로 관리한다.
- 모든 멤버가 반드시 로그인 계정을 가져야 하는 것은 아니다.
- 로그인 후 접근 권한은 `role` 기준으로 처리한다.

## 핵심 엔티티

### user

- 로그인 가능한 계정
- 시스템 접근 권한 보유
- 역할:
  - `member`
  - `professor`
  - `admin`

### member

- 공개 프로필 또는 연구실 구성원 정보
- 홈페이지에 노출되는 사람 정보
- 로그인 계정과 1:1 또는 선택적 연결

원칙:

- `member`는 있어도 `user`가 없을 수 있다.
- `user`는 보통 하나의 `member`와 연결된다.
- 관리자 계정은 `member` 없이 존재할 수도 있다.

## 로그인 방식

초기 권장:

- 이메일 기반 로그인
- 필요 시 OAuth 추가

MVP 기준 추천:

- `학교/연구실 이메일 기반 로그인`
- 초기에는 관리자 수동 승인 또는 수동 계정 생성

확장 가능:

- Google OAuth
- GitHub OAuth

초기 목표는 `편의성`보다 `권한 통제`다.

## 계정 생성 원칙

### 기본 방식

- 공개 회원가입은 바로 열지 않는다.
- 관리자가 내부 인원 계정을 생성하거나 승인한다.

이유:

- 내부 포털 접근 권한이 있기 때문
- 외부 사용자의 무분별한 가입을 막아야 함

### 추천 흐름

1. 관리자가 사용자 계정 생성
2. 멤버 프로필과 계정 연결
3. 역할 지정
4. 사용자가 첫 로그인 후 비밀번호 설정 또는 프로필 확인

## 역할 설계

- `public`
  - 비로그인 사용자
- `member`
  - 내부 포털 접근 가능
- `professor`
  - 내부 포털 + 교수님 전용 기능 접근 가능
- `admin`
  - 전체 계정 및 콘텐츠 관리 가능

## 계정 상태 모델

역할과 별개로, 계정은 상태를 가진다.

추천 상태:

- `invited`
  - 관리자가 계정을 만들었지만 아직 첫 로그인을 하지 않음
- `active`
  - 정상 사용 가능
- `inactive`
  - 일시적으로 비활성화됨
- `suspended`
  - 권한 문제 또는 운영상 사유로 정지
- `archived`
  - 졸업 또는 종료 후 보관 상태

운영 원칙:

- `role`은 권한
- `account_status`는 사용 가능 상태
- 두 값을 분리해서 관리한다

예시:

- `member + active`
- `member + archived`
- `professor + active`
- `admin + active`

## 로그인 후 리다이렉트 규칙

- `member`
  - 기본 이동: `/internal`
- `professor`
  - 기본 이동: `/professor`
  - 필요 시 `/internal`도 접근 가능
- `admin`
  - 기본 이동: `/internal`
  - 운영 메뉴에서 관리자 기능 접근

공통 규칙:

- 사용자가 보호된 페이지를 보려다 로그인하면, 로그인 후 원래 URL로 복귀
- 원래 URL이 없으면 역할별 기본 경로로 이동

## 첫 로그인 플로우

추천 흐름:

1. 관리자가 계정 생성
2. 사용자는 발급받은 계정으로 로그인
3. 첫 로그인 여부 확인
4. 비밀번호 변경 강제
5. 계정 요약 또는 연결된 멤버 프로필 확인
6. 역할에 따라 기본 페이지로 이동

UI 원칙:

- 첫 로그인 사용자는 일반 로그인 후 바로 내부 홈으로 보내지 않는다
- 먼저 `비밀번호 변경 + 계정 확인` 단계를 거친다
- 불필요한 설정은 첫 로그인에 몰아넣지 않는다

첫 로그인 완료 후 이동:

- `member` -> `/internal`
- `professor` -> `/professor`
- `admin` -> `/internal`

## 계정 페이지 `/account`

목적:

- 사용자가 자신의 로그인 계정 상태와 연결된 프로필을 확인하는 페이지

표시 항목:

- 이름
- 이메일
- 역할
- 연결된 멤버 프로필
- 프로필 이미지
- 최근 로그인 시간

가능 액션:

- 표시 이름 수정
- 프로필 이미지 변경
- 비밀번호 변경 또는 재설정 진입
- 연결된 멤버 프로필 보기

초기에는 넣지 않아도 되는 것:

- 복잡한 알림 설정
- 다중 계정 연결
- 세부 개인정보 설정

## 관리자 계정 발급 플로우

추천 흐름:

1. 관리자 계정 생성
2. 기본 역할 선택
3. 연결할 `member` 선택 또는 미연결 상태로 저장
4. 임시 비밀번호 발급
5. 계정 상태를 `invited`로 저장
6. 사용자에게 계정 정보 전달
7. 첫 로그인 후 상태를 `active`로 전환

관리자 화면에서 보이면 좋은 항목:

- 이름
- 이메일
- 역할
- 계정 상태
- 연결된 멤버
- 마지막 로그인 시간

초기 UX 원칙:

- 한 화면에서 `계정 생성 + 역할 지정 + 멤버 연결`이 끝나야 한다
- 너무 많은 설정을 한 번에 받지 않는다
- 상태 배지는 항상 눈에 잘 띄어야 한다

## 관리자 계정 관리 범위

관리자 전용 기능 예시:

- 사용자 목록 보기
- 사용자 검색
- 역할 변경
- 활성 / 비활성 처리
- `user`와 `member` 연결
- 계정 생성

초기 MVP에서는:

- 별도 관리자 페이지 없이 DB 또는 간단한 내부 관리 화면으로 운영 가능

중장기:

- `/admin/users`
- `/admin/members-linking`

## 상태별 화면 정책

### `invited`

- 로그인 가능
- 첫 로그인 플로우 강제
- 일반 내부 화면 직접 진입 제한 가능

### `active`

- 정상 사용 가능

### `inactive`

- 로그인은 막거나, 로그인 후 사용 제한 안내 표시

### `suspended`

- 로그인 차단
- 관리자 문의 문구 표시

### `archived`

- 기본적으로 로그인 차단
- 멤버 프로필은 공개 상태에 따라 유지 가능

## 권한 부족 / 계정 상태 UX

구분해서 보여줘야 하는 상태:

- 로그인 필요
- 권한 없음
- 비활성 계정
- 정지된 계정
- 첫 로그인 미완료

메시지 원칙:

- 왜 접근이 안 되는지 짧게 설명
- 다음 행동을 안내
- 관리자 문의 방법을 함께 표시

예시:

- `이 페이지는 연구실 내부 계정으로만 접근할 수 있습니다.`
- `현재 계정은 아직 활성화되지 않았습니다. 관리자에게 문의하세요.`
- `첫 로그인 설정을 완료한 뒤 계속 진행할 수 있습니다.`

## 세션 및 권한 체크 원칙

- 세션은 서버에서 확인한다.
- 클라이언트는 세션 표시용으로만 사용한다.
- 권한 없는 사용자는 메뉴를 숨기는 것과 별개로 라우트 접근도 막는다.
- 데이터도 서버에서 권한 필터링 후 전달한다.

## 보호 라우트 규칙

- `/account`
  - 로그인 사용자만 접근 가능
- `/internal/*`
  - `member`, `professor`, `admin`
- `/professor/*`
  - `professor`, `admin`

## `user`와 `member` 연결 규칙

추천 필드:

- `users.member_id`

원칙:

- 로그인 후 계정 페이지에서 연결된 멤버 정보를 확인 가능
- 멤버 프로필이 없는 계정도 허용
- 교수님은 보통 `member` 프로필과 연결

## 최소 구현 범위

MVP에서 꼭 필요한 것:

- 로그인
- 세션 유지
- 역할 기반 리다이렉트
- `/account` 페이지
- 내부 / 교수님 라우트 보호

MVP 이후:

- 관리자 사용자 관리 화면
- OAuth 확장
- 비밀번호 정책 강화

## 코딩 에이전트에게 줄 핵심 지시

- 인증은 `Supabase Auth`로 구현 (`supabase.auth.signInWithPassword`, `supabase.auth.signUp` 등 사용)
- `user`와 `member`는 분리
- 회원가입은 공개 오픈하지 않음
- 계정은 관리자 발급 방식으로 운영
- `role`과 `account_status`를 분리
- 첫 로그인 시 비밀번호 변경 플로우 적용
- 로그인 후 역할별 리다이렉트 적용
- `/account`는 최소한 계정 요약 페이지로 구현
- 권한 체크는 서버 기준으로 처리
