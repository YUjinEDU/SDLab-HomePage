# 전략 및 기술 방향

상위 문서: [[홈페이지 계획]]

## 현재 상태

- **Phase 1 (공개 포털 MVP): 완료** — 96% 디자인 매치율, 15/15 기능 요건 충족
- 7개 공개 페이지 + 상세 페이지 구현 완료 (Home, Members, Students, Research, Publications, Projects, Patents, Contact)
- 67+ TypeScript 파일, 정적 데이터 파일(data/\*.ts) 기반 MVP 운영 중
- Phase 2 (인증 + 내부 포털) 준비 중

이 문서의 역할:

- 왜 이 프로젝트를 `Next.js 기반 공개 포털 + 내부 포털` 구조로 가는지 설명하는 전략 문서

## 결론

이 프로젝트는 `WordPress 기반 홈페이지`보다 `Next.js 기반 커스텀 웹사이트`로 구축하는 것이 맞다.

이유:

- 요구사항이 단순 소개 사이트가 아니라 `공개 사이트 + 내부 포털` 구조다.
- `로그인`, `권한별 페이지 접근`, `프로젝트 데모`, `GPU 모니터링`, `교수님 전용 자동화 페이지`, `내부 캘린더`가 필요하다.
- WordPress는 초기 외형은 빠르지만 기능이 늘수록 플러그인 의존성이 커질 가능성이 있다.
- 코딩 에이전트를 적극 활용할 계획이므로 코드 기반 확장성이 중요하다.

## 핵심 원칙

- `디자인은 템플릿을 활용`
- `구현은 직접 개발`
- `인증/DB/배포는 관리형 서비스 활용`

즉, `WordPress 템플릿`이 아니라 `Next.js 프론트엔드 템플릿 또는 UI starter`를 사용한다.

## 참고 레퍼런스 해석

참고 문서: [[참고 레퍼런스]]

- `ISoftLab`
  - 안정적이고 단정한 연구실 사이트 구조
  - Members / Research / Publications / Board / Contact 패턴이 명확함

우리 사이트는 `ISoftLab의 안정성 + suanlab의 확장성` 방향으로 간다.

## 최종 기술 방향

### 기본 스택

- 프레임워크: `Next.js 16` (App Router, Turbopack)
- 스타일링: `Tailwind CSS v4` (@theme inline 사용)
- UI 컴포넌트: shadcn/ui 미사용 — `순수 Tailwind CSS utility-first` 방식으로 구현
- 패키지 매니저: `pnpm`
- 배포: `Vercel`

### 인증 / 데이터

- 인증: `Supabase Auth` (Auth.js에서 변경)
- 데이터 저장: `Supabase` (PostgreSQL)
- 권한 구조: `public / member / professor / admin`

### 콘텐츠 관리

초기:

- 교수 소개
- 멤버 정보
- 논문 목록
- 프로젝트 정보
- 공지사항

중장기:

- 필요 시 headless CMS 연동
- 후보: `Sanity`, `Payload`

## 템플릿 전략

### 왜 템플릿을 쓰는가

- 초기 외형 제작 시간을 줄인다.
- 교수님께 빠르게 시안을 보여줄 수 있다.
- 레퍼런스 수준의 레이아웃을 빠르게 확보할 수 있다.

### 어떤 템플릿을 쓸 것인가

추천 방식:

1. `Next.js 16 기본 프로젝트`로 시작
2. `Tailwind CSS v4 utility-first`로 UI 구성 (shadcn/ui 미사용)
3. 공개 페이지는 기관형 / 마케팅형 레이아웃으로 구성
4. 내부 페이지는 대시보드 스타일로 별도 구성

## 유지보수 원칙

- 인증을 직접 구현하지 않는다.
- 공통 UI는 컴포넌트화한다.
- 공개 영역과 내부 영역을 라우트 기준으로 분리한다.
- 멤버 / 논문 / 프로젝트 데이터 구조를 초기에 확정한다.
- 기능 하나를 추가할 때 페이지, 권한, 데이터 구조를 같이 정리한다.
