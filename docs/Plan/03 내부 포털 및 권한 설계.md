# 내부 포털 및 권한 설계

상위 문서: [[홈페이지 계획]]

이 문서의 역할:

- 내부 포털과 교수님 전용 영역의 권한, 라우트 보호, 내부 UX를 정의하는 문서

## 내부 영역 구조

- `/internal`
- `/internal/calendar`
- `/internal/gpu`
- `/internal/resources`
- `/internal/projects`

## 교수님 전용 영역

- `/professor`
- `/professor/lecture`

## 역할 정의

- `public`
- `member`
- `professor`
- `admin`

## 내부 포털 설계 목표

- 공개 홈페이지와 분리된 `실용적 작업 공간`을 제공한다.
- 내부 구성원이 매일 접속해도 불편하지 않은 정보 밀도를 유지한다.
- 권한별로 메뉴와 데이터가 자연스럽게 달라지도록 설계한다.
- 교수님 전용 기능은 별도 영역으로 분리해 공개/내부 기능과 섞이지 않게 한다.

## 기본 권한 원칙

- `public`
  - 공개 페이지만 접근 가능
- `member`
  - 내부 포털, 캘린더, GPU 모니터링, 내부 자료 접근 가능
- `professor`
  - member 권한 포함
  - 교수님 전용 자동화 기능 접근 가능
- `admin`
  - 전체 데이터 생성/수정/삭제 가능

## 인증 및 세션 원칙

- 로그인하지 않은 사용자는 `/internal` 및 `/professor` 하위 라우트에 접근할 수 없다.
- 세션은 서버 기준으로 검증한다.
- 메뉴 노출만 숨기지 않고, 라우트 접근 자체를 차단한다.
- 권한 체크는 `layout` 또는 공통 서버 유틸에서 일관되게 처리한다.
- 비공개 데이터는 클라이언트에서 필터링하지 않고 서버에서 먼저 제한한다.

## 권한 체크 단위

권한은 아래 4단계에서 검사한다.

1. 라우트 단위

- 페이지 접근 가능 여부

2. 메뉴 단위

- 어떤 메뉴를 보여줄지

3. 데이터 단위

- 어떤 레코드를 읽을 수 있는지

4. 액션 단위

- 생성 / 수정 / 삭제 / 실행 가능 여부

예시:

- `member`는 내부 일정 읽기 가능
- `member`는 교수님 강의 자동화 실행 불가
- `admin`은 모든 게시글 수정 가능
- `professor`는 강의 자동화 실행 가능하지만 전체 시스템 설정 변경은 제한 가능

## 역할별 상세 권한

### public

가능:

- 공개 페이지 열람
- 공개 데모 접근
- 공개 공지 검색

불가:

- 내부 포털 접근
- 내부 캘린더 열람
- GPU 상태 열람
- 내부 자료실 접근
- 교수님 전용 도구 접근

### member

가능:

- 공개 페이지 열람
- 내부 포털 접근
- 내부 일정 열람
- GPU 모니터링 열람
- 내부 자료 열람
- 내부 데모 중 member 허용 항목 접근

제한:

- 교수님 전용 도구 접근 불가
- 전체 사이트 관리 불가

### professor

가능:

- member 권한 전체
- 교수님 전용 자동화 페이지 접근
- 교수님 전용 작업 실행 이력 조회

제한:

- 관리자 전용 시스템 설정은 별도 분리 가능

### admin

가능:

- 전체 페이지 접근
- 전체 데이터 생성/수정/삭제
- 권한 관리
- 운영 도구 접근

## 내부 포털 레이아웃 설계

공통 구조:

- 상단 바
  - 현재 섹션 제목
  - 검색 또는 빠른 이동
  - 사용자 메뉴
- 좌측 사이드바
  - Internal Home
  - Calendar
  - GPU Monitor
  - Resources
  - Internal Projects
- 본문 영역
  - 카드 / 테이블 / 리스트 중심

디자인 원칙:

- 공개 사이트보다 밀도 높게
- 카드형 정보와 표형 정보를 혼합
- 우선순위 높은 상태 정보는 상단 배치
- 모바일에서는 사이드바를 드로어로 전환

## 내부 포털 상세

### Internal Home `/internal`

구성:

- 오늘 일정 요약
- 최근 공지
- 내부 링크 모음
- GPU 상태 요약
- 자주 쓰는 도구 바로가기

권장 컴포넌트:

- `InternalHeader`
- `TodayScheduleCard`
- `RecentInternalBoardCard`
- `GpuStatusSummaryCard`
- `QuickLinksCard`
- `PinnedResourcesCard`

핵심 목적:

- 내부 인원이 로그인 후 가장 먼저 보는 대시보드
- 오늘 필요한 정보만 빠르게 확인하는 페이지

### Internal Calendar `/internal/calendar`

구성:

- 월간 / 주간 뷰
- 연구실 세미나
- 과제 일정
- 교수님 일정 공유 항목

권한:

- `member`, `professor`, `admin`

권장 컴포넌트:

- `CalendarToolbar`
- `CalendarViewSwitcher`
- `EventListPanel`
- `UpcomingDeadlinesCard`
- `CalendarEventModal`

핵심 액션:

- 일정 조회
- 일정 상세 확인
- 권한 있는 사용자만 일정 생성/수정

운영 메모:

- 초기에는 읽기 중심으로 시작하고, 입력 기능은 2차로 붙여도 된다

### GPU Monitor `/internal/gpu`

구성:

- GPU 서버 목록
- 사용률
- 메모리 사용량
- 프로세스 정보
- 서버별 상태 카드
- 외부 모니터링 시스템 링크

권한:

- `member`, `professor`, `admin`

비고:

- 초기에는 iframe 또는 외부 링크 형태로 연결하고, 이후 자체 대시보드화 가능

권장 컴포넌트:

- `GpuServerCard`
- `GpuUsageChart`
- `GpuProcessTable`
- `ServerHealthBadge`
- `ExternalMonitorLinkCard`

핵심 목적:

- 연구실 구성원이 현재 자원 상태를 한눈에 확인
- 빠르게 어떤 서버를 사용할 수 있는지 판단

실행 전략:

- 1차: 외부 모니터링 링크 / iframe
- 2차: API 연동 대시보드
- 3차: 사용자별 잡 상태 또는 예약 현황 연동

## 접근 실패 UX

### 비로그인 사용자

- `/login`으로 이동
- 로그인 후 원래 보려던 페이지로 복귀

### 권한 부족 사용자

- `접근 권한이 없습니다` 안내 페이지 노출
- 필요한 경우 관리자 또는 교수님에게 요청하라는 안내 문구 표시

### 데이터 없음

- `표시할 데이터가 없습니다`
- 첫 설정이 필요한 경우 관리자 액션 버튼 제공

## 콘텐츠별 권한 예시

- `members`
  - 공개 프로필: `public`
  - 내부 연락처 세부정보: `member`
- `publications`
  - 대부분 `public`
- `projects`
  - 공개 설명은 `public`
  - 내부 메모와 산출물 링크는 내부 엔티티로 분리
- `demos`
  - 공개/비공개를 `visibility`로 분리
- `board_posts`
  - 대외 공지: `public`
  - 내부 공지: `member`
- `resources`
  - 기본적으로 `member`
- `calendar_events`
  - 기본적으로 `member`
- `professor_tools_runs`
  - `professor`, `admin`
