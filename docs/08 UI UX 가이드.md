상위 문서: [[홈페이지 계획]]

관련 문서:

- [[07 인증 및 계정관리 설계]]
- [[참고 레퍼런스]]

이 문서의 역할:

- 공개 포털과 내부 포털의 전체 UI/UX 기준과 디자인 시스템 최소 규칙을 정의하는 문서

목적:

- 공개 사이트와 내부 포털의 경험을 일관되게 설계한다.
- 코딩 에이전트가 단순 구현이 아니라 `좋은 화면 구조`로 작업하도록 기준을 준다.
- 페이지마다 달라지는 UI를 막고, 공통 패턴을 먼저 고정한다.

## 1. 전체 방향

이 사이트는 두 가지 성격을 동시에 가져야 한다.

- 공개 사이트
  - 연구실 소개와 대외 신뢰 형성
- 내부 포털
  - 실제 업무와 운영 편의를 위한 실용 도구

따라서 하나의 브랜드 안에서 `두 가지 화면 톤`을 구분한다.

### 공개 사이트 톤

- 깔끔함
- 신뢰감
- 절제된 현대성
- 연구 성과 중심

### 내부 포털 톤

- 정보 밀도 높음
- 빠른 확인
- 상태 중심
- 도구형 인터페이스

## 2. 시각 방향

기본 원칙:

- 전체 배경은 밝게 유지
- 포인트 컬러는 그린 계열 중심
- 텍스트는 진한 네이비 또는 딥 그레이
- 과한 그라데이션, 과한 모션, 과한 장식은 피한다

추천 톤:

- 공개 사이트: `밝은 배경 + 넉넉한 여백 + 그린 포인트 + 카드형 정보 구조`
- 내부 포털: `밝은 배경 + 구획이 분명한 패널 + 딥그린 상태 강조`

피해야 할 것:

- 지나치게 스타트업 랜딩 페이지 같은 과장된 히어로
- 너무 무거운 다크 테마 중심 구성
- 페이지마다 다른 카드 디자인

### 그린 브랜딩 원칙

- 메인 포인트는 `차분한 연구실 그린`으로 잡는다
- 너무 형광색에 가까운 그린은 피한다
- 그린은 `강조`, `상태`, `브랜드 포인트`에 주로 사용한다
- 큰 면적은 연한 그린 틴트로 사용하고, 진한 그린은 버튼/배지/강조선에 사용한다

추천 감각:

- 메인 그린: 차분한 에메랄드 또는 포레스트 그린
- 서브 틴트: 아주 연한 민트/세이지 계열
- 중립 배경: 오프화이트, 라이트 그레이
- 본문 텍스트: 슬레이트/차콜

### 공개 사이트에서의 그린 사용

- 네비게이션 활성 상태
- 주요 CTA 버튼
- 링크 hover
- 태그 및 배지 일부
- 섹션 제목의 강조 단어

원칙:

- 공개 사이트에서는 그린을 `브랜드 인상`으로 쓴다
- 한 화면에 진한 그린 면적이 너무 많지 않게 조절한다

### 내부 포털에서의 그린 사용

- 상태 요약 카드
- 성공 / 사용 가능 / 정상 상태 표시
- 현재 선택된 메뉴 강조
- 주요 실행 버튼

원칙:

- 내부 포털에서는 그린을 `브랜드`보다 `상태와 행동`에 더 강하게 쓴다
- 경고와 오류는 그린과 명확히 구분되는 별도 색상 사용

## 3. 정보 위계

모든 페이지는 아래 우선순위를 명확히 가져야 한다.

1. 이 페이지가 무엇인지
2. 지금 사용자가 뭘 할 수 있는지
3. 가장 중요한 정보가 무엇인지
4. 그다음 탐색 가능한 정보가 무엇인지

예시:

### 홈

1. 연구실 정체성
2. 핵심 연구 분야
3. 대표 성과
4. 최근 소식

### 내부 홈

1. 오늘 일정
2. 중요 공지
3. GPU 상태
4. 자주 쓰는 링크

### Publications

1. 필터
2. 통계
3. 논문 카드 목록

## 4. 공통 패턴

페이지마다 다른 UX를 만들지 말고, 아래 패턴을 최대한 통일한다.

### 검색 패턴

- 검색창은 항상 리스트 상단
- placeholder 문구는 짧고 구체적으로
- 검색 결과 수는 바로 아래에 표시

예시:

- `논문 제목, 저자, 키워드 검색`
- `프로젝트명, 기관명 검색`

### 필터 패턴

- 검색창 아래에 필터 배치
- 주요 필터만 기본 노출
- 나머지는 접기 가능
- 선택된 필터는 chip 형태로 보여줌

### 카드 패턴

모든 카드 공통 구조:

- 상단: 상태 / 카테고리 / 메타
- 중단: 제목
- 하단: 요약 정보
- 마지막: 주요 액션 버튼

### 상세 페이지 패턴

- Header
- 핵심 메타 정보
- 본문
- 관련 콘텐츠

## 5. 페이지별 UX 원칙

### 홈

- 처음 방문한 사용자가 5초 안에 연구실 정체성을 이해해야 한다
- 연구 분야, 대표 성과, 최근 소식 순서가 자연스러워야 한다
- CTA는 많아도 2개를 넘기지 않는다

### Members

- 그룹 이동이 쉬워야 한다
- 한 사람의 프로필에서 논문과 프로젝트로 쉽게 이동 가능해야 한다
- 졸업생은 너무 무겁지 않게 축약해서 보여준다

### Publications

- 필터가 핵심이다
- 논문 카드는 길어도 구조가 흔들리면 안 된다
- `Copy BibTeX`, `Copy reference`, `Open PDF`의 액션 위치를 통일한다

### Projects

- 상태, 기관, 기간이 한눈에 보여야 한다
- 연구 과제와 데모 연결이 쉬워야 한다
- 카테고리 패널은 탐색을 돕는 수준이어야지 화면을 압도하면 안 된다

### Board

- 검색과 최신성 중심
- 제목과 날짜 가독성이 중요
- 공지, 수상, 논문 채택 같은 카테고리는 시각적으로 구분되어야 한다

### Live Demo

- `무엇을 하는 데모인지`를 한 줄로 이해할 수 있어야 한다
- 공개 데모와 내부 데모는 배지로 명확히 구분한다
- 액션은 `실행`, `문서`, `관련 프로젝트` 정도로 단순화한다

### Contact

- 정보가 많기보다 신뢰감 있고 정확해야 한다
- 위치, 이메일, 문의 방식이 바로 보여야 한다

## 6. 내부 포털 UX 원칙

### 내부 홈

첫 화면 우선순위:

1. 오늘 일정
2. 중요 공지
3. GPU 상태 요약
4. 자주 쓰는 링크
5. 최근 자료

### Calendar

- 읽기 쉬운 월간/주간 전환
- 일정 생성보다 일정 확인이 먼저
- 중요한 일정은 색상과 아이콘으로 구분

### GPU Monitor

- 숫자보다 상태 판단이 먼저 가능해야 한다
- `사용 가능 / 혼잡 / 장애`처럼 빠른 해석이 가능해야 한다
- 차트는 보조 정보로 사용한다

### Resources

- 검색이 가장 중요
- 자주 보는 자료는 상단 고정
- 링크형 자료와 문서형 자료를 구분

### Professor Tools

- 복잡한 자동화 흐름을 단순한 단계형 UI로 풀어야 한다
- 입력, 실행, 결과, 로그를 한 화면에 다 때려넣지 않는다
- 실행 버튼은 가장 명확하게 강조

## 7. 상태 화면 설계

정상 화면만 설계하면 실제 UX가 약해진다. 아래 상태를 반드시 고려한다.

### Empty State

- 데이터가 없을 때 무엇이 비어 있는지 설명
- 관리자나 권한 있는 사용자는 다음 행동 버튼 제공

예시:

- `아직 등록된 내부 자료가 없습니다.`

### No Results

- 검색 결과가 없을 때 현재 필터를 보여주고, 초기화 버튼 제공

예시:

- `검색 결과가 없습니다. 필터를 줄이거나 검색어를 바꿔보세요.`

### Access Denied

- 권한 부족 이유를 짧게 설명
- 가능한 경우 관리자 문의 유도

### Loading

- 긴 로딩은 skeleton 또는 상태 메시지 사용
- 표, 카드, 차트는 각각 다른 로딩 UI를 갖는 것이 좋다

### Error State

- 기술적 에러 메시지 대신 사용자 행동 중심 문구 사용

예시:

- `데이터를 불러오지 못했습니다. 잠시 후 다시 시도하세요.`

## 8. 계정 UX 원칙

- 로그인 페이지는 단순해야 한다
- 첫 로그인 사용자는 일반 사용자와 다른 플로우를 사용한다
- `/account`는 설정 페이지보다 `내 계정 요약`에 가깝게 설계한다
- 계정 상태가 정상적이지 않을 때는 이유와 다음 행동을 분명히 보여준다

## 9. 디자인 시스템 최소 기준

고정해야 하는 것:

- 색상 토큰
- 타이포그래피 스케일
- spacing 규칙
- 카드 반경
- 버튼 스타일
- 배지 스타일
- 상태 색상

### 추천 색상 토큰 방향

- `primary`
  - 메인 그린
- `primary-foreground`
  - 흰색 또는 매우 밝은 중립색
- `primary-muted`
  - 연한 그린 틴트
- `accent`
  - 세이지 또는 민트 계열
- `background`
  - 오프화이트
- `surface`
  - 화이트
- `border`
  - 옅은 그레이-그린 계열 가능

### 상태 색상 예시

- `active`
  - green
- `completed`
  - gray
- `warning`
  - amber
- `error`
  - red
- `internal`
  - deep green or neutral dark badge
- `info`
  - teal or muted cyan

### 버튼 원칙

- Primary 버튼은 그린 계열
- Secondary 버튼은 흰 배경 + 그린 보더
- Ghost 버튼은 중립 톤
- 위험 액션만 red 계열 사용

### 배지 원칙

- 카테고리 배지는 연한 배경 + 진한 텍스트
- 상태 배지는 의미가 바로 읽히게 색상 차이를 둔다
- public / internal / professor 같은 접근 범위 배지는 형태와 색을 함께 구분한다

## 10. 코딩 에이전트에게 줄 기준

- 페이지마다 새로운 UI를 발명하지 않는다
- 검색, 필터, 카드, 상세 페이지 구조를 공통 패턴으로 맞춘다
- 공개 사이트는 브랜딩 중심, 내부 포털은 실용 중심으로 구분한다
- 상태 화면을 반드시 구현한다
- 액션 버튼 수를 최소화한다

한 줄 원칙:

`보기에 예쁜 것보다, 처음 보는 사람이 빠르게 이해하고 바로 행동할 수 있는 UI를 만든다.`
