Apply these changes across ALL action files:

**A. Replace all @ts-expect-error revalidateTag calls:**

- Import `revalidate` from `@/lib/utils/revalidate` instead of `revalidateTag` from `next/cache`
- Replace all `// @ts-expect-error ... \n revalidateTag("xxx")` with `revalidate("xxx")`
- Affected files: publications.ts (3 instances), projects.ts (3 instances), visibility.ts (2 instances)

**B. Standardize return types to ActionResult:**

- Import `ActionResult` from `@/types` in each action file
- Update function return types to `Promise<ActionResult>` (for non-redirect actions)
- Ensure all success paths return `{ success: true }` and all error paths return `{ error: "message" }`
- For auth.ts login: the function calls `redirect()` on success which throws (never returns), so the return type stays as-is but error returns should match `{ error: string }`

**C. Fix login error enumeration (auth.ts):**

- Replace `return { error: error.message }` with `return { error: "이메일 또는 비밀번호가 올바르지 않습니다" }` — a generic message regardless of the specific Supabase error

**D. Add basic input validation to each action:**

- At minimum, validate that required string fields from `formData.get()` are non-null and non-empty after trim
- For numeric fields (year, displayOrder), validate `parseInt` result is not `NaN`
- Return `{ error: "필수 항목을 입력해주세요" }` (or a field-specific message) if validation fails
- This is manual validation (not zod) — per CONTEXT.md, full zod schemas are deferred
- Key files: contact.ts (lab_name_ko required), members.ts (name_ko, email required, displayOrder NaN check), publications.ts (title, year required, year NaN check), projects.ts (title required), news.ts (title required)

**E. Fix non-atomic join-table mutations (news.ts, projects.ts, publications.ts):**

- In `createNews`, `updateNews`, `createProject`, `updateProject`, `createPublication`: after the join-table insert, check for errors. If a join-table insert fails, return `{ error: "관련 항목 연결에 실패했습니다: " + joinError.message }`
- Do NOT change `updatePublication` which already uses an RPC for atomicity

**F. Remove redundant CASCADE deletes (news.ts):**

- In `deleteNews`: remove the manual `await supabase.from("news_projects").delete()...` and `await supabase.from("news_publications").delete()...` calls before the parent delete. The schema defines `ON DELETE CASCADE` for these join tables, so they are cleaned up automatically.
- Keep the manual join-table deletes in `updateNews` since CASCADE does not apply during updates.

**G. Standardize all return shapes:**
After all changes, every server action should return either:

- `{ success: true }` on success (before any redirect/revalidate)
- `{ error: "descriptive message" }` on failure
- Or call `redirect()` which throws (login success case)
