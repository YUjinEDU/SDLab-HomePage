2. **Profile update uses member_id FK (profile.ts)**:
In `updateMyProfile`, replace the email-based member lookup:
```typescript

// .from("members").select("id, slug").eq("email", user.email!).single()

const { data: profile } = await supabase
  .from("profiles")
  .select("member_id")
  .eq("id", user.id)
  .single();

if (!profile?.member_id) return { error: "연결된 멤버 정보를 찾을 수 없습니다" };

  .from("members")

  .single();
````

This uses the `profiles.member_id` FK instead of matching by email, which is more robust.

3. **Middleware profile query deduplication (middleware.ts)**:
   Refactor the middleware to query profiles once after `getUser()` succeeds, then reuse the result for both `/professor` and `/login` path checks:

   ```typescript
   const {
     data: { user },
   } = await supabase.auth.getUser();

   // Query profile once if user is authenticated
   let userRole: string | null = null;
   if (user) {
     const { data: profile } = await supabase
       .from("profiles")
       .select("role")
       .eq("id", user.id)
       .single();
     userRole = profile?.role ?? null;
   }

   // Use userRole for both /professor and /login checks
   if (user && request.nextUrl.pathname.startsWith("/professor")) {
     if (userRole !== "professor" && userRole !== "admin") {
       return NextResponse.redirect(new URL("/internal", request.url));
     }
   }

   if (request.nextUrl.pathname === "/login" && user) {
     const target =
       userRole === "professor" || userRole === "admin"
         ? "/professor"
         : "/internal";
     return NextResponse.redirect(new URL(target, request.url));
   }
   ```


**Option B (if no generated types — more likely):**
Define a minimal row type inline that matches the SELECT columns:

```typescript
type MemberRow = {
  id: string;
  name_ko: string;
  name_en: string | null;
  email: string | null;
  // ... all columns used in the mapper
};

function toMember(row: MemberRow): Member { ... }
```

Remove all `as string`, `as number`, `as boolean` casts from the mapper bodies. The typed row parameter makes these unnecessary. If a field could be null in the DB but the domain type expects non-null, use nullish coalescing: `row.name_en ?? ""`.

2. **Consistent error handling in single-item queries:**

   In single-item queries (getPublicationBySlug, getProjectBySlug, getMemberBySlug, etc.), distinguish between "not found" and real errors:

   ```typescript
   const { data, error } = await supabase
     .from("table")
     .select("*")
     .eq("slug", slug)
     .single();

   if (error) {
     // PGRST116 = "JSON object requested, multiple (or no) rows returned"
     if (error.code === "PGRST116") return null; // genuinely not found
     console.error("Unexpected query error:", error);
     return null; // or throw, depending on preference — but log it
   }
   return data ? toMapper(data) : null;
   ```

   This ensures real DB errors (connection failures, RLS issues) are logged instead of silently treated as "not found".
